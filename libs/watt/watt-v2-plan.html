1. Summary

Build Watt v2 as side-by-side next entrypoints inside the existing @energinet/watt package.

V2 will use Angular Signal Forms only, design tokens synced from Figma Enterprise Variables API, browser-native APIs instead of Angular Material/CDK, and Storybook as the reference for existing features and visual parity.

2. Package Strategy

Use new next entrypoints:

  • @energinet/watt/next/button
  • @energinet/watt/next/text-field
  • @energinet/watt/next/dropdown
  • @energinet/watt/next/date-field
  • @energinet/watt/next/styles
  • @energinet/watt/next/tokens

Keep current v1 entrypoints untouched:

  • @energinet/watt/button
  • @energinet/watt/text-field

This allows component-by-component migration without breaking existing applications.

5. Design Token Sync

Figma Enterprise Variables API is the source of truth.

Primary endpoint:

GET /v1/files/{FIGMA_FILE_KEY}/variables/local

Required token scope: file_variables:read

Configure: Replace {FIGMA_FILE_KEY} with the Figma file key from your project configuration.

Optional controlled write/bootstrap endpoint:

  • POST /v1/files/{FIGMA_FILE_KEY}/variables
  • Optional write scope: file_variables:write

6. Token Sync Flow

1. Figma Variables
2. Normalized token JSON
3. Generated CSS custom properties
4. Optional SCSS metadata
5. Optional TypeScript metadata
6. Storybook token documentation

Generated files

  • libs/watt/package/next/tokens/tokens.json
  • libs/watt/package/next/core/styles/tokens.css
  • libs/watt/package/next/core/styles/tokens.scss
  • libs/watt/package/next/core/tokens/tokens.ts

Recommended token collections

  • core: raw colors, typography, spacing, radius, shadows, sizing
  • semantic: text, surface, border, focus, disabled, danger, warning, success, info
  • component: button, field, dropdown, card, modal, drawer, toast, etc.

Recommended naming

  • color/primary/500
  • color/neutral/900
  • text/on-light/high-emphasis
  • space/m
  • shadow/drawer
  • button/primary/background/default
  • field/border/error

Recommended CSS variables

  • --watt-next-color-primary-500
  • --watt-next-text-high-emphasis
  • --watt-next-space-m
  • --watt-next-shadow-drawer
  • --watt-next-button-primary-background

7. Token Automation

Add Nx targets:

  • nx run watt:tokens:sync
  • nx run watt:tokens:check

tokens:sync should

  • Fetch Figma variables.
  • Normalize variable names, modes, aliases, and values.
  • Generate committed token artifacts.
  • Update Storybook token metadata.

tokens:check should

  • Fetch Figma variables.
  • Regenerate token artifacts.
  • Fail if generated files differ from committed files.

GitHub workflow

  • Manual trigger.
  • Scheduled trigger.
  • Uses FIGMA_TOKEN secret.
  • Runs token sync.
  • Opens PR if generated files changed.

8. Token Sync Rules

  • Figma owns token values.
  • Code owns generated artifacts.
  • Do not manually edit generated token files.
  • Use one-way sync from Figma to code for normal work.
  • Avoid routine code-to-Figma sync.
  • Use code-to-Figma only for controlled bootstrap or migration work.
  • Component CSS must consume tokens, not hardcoded Figma values.

Example component CSS should use token variables such as:

  • background: var(--watt-next-button-primary-background)
  • color: var(--watt-next-button-primary-foreground)
  • border-color: var(--watt-next-button-primary-border)

11. Signal Forms Strategy

All v2 form controls use Angular Signal Forms only.

Use FormValueControl<T> for

  • Text field
  • Textarea
  • Dropdown
  • Search field
  • Date field
  • Time field
  • Datetime field
  • Year field
  • Year-month field
  • Input chip
  • DropZone

Use FormCheckboxControl for

  • Checkbox
  • Slide toggle

Do not use

  • ControlValueAccessor
  • ngModel
  • Reactive Forms compatibility wrappers
  • Template-driven forms

Recommended control shape

  • value = model<T | null>(null)
  • touched = model(false)
  • disabled = input(false)
  • readonly = input(false)
  • required = input(false)
  • invalid = input(false)
  • errors = input([])

Validation rules stay in the Signal Forms schema.

Components display validation state but do not own validation logic.

12. Date And Time Strategy

Use Temporal for all date/time values.

  • Date field: Temporal.PlainDate
  • Time field: Temporal.PlainTime
  • Datetime field: Temporal.PlainDateTime
  • Year-month field: Temporal.PlainYearMonth, if suitable
  • Absolute timestamp: Temporal.Instant
  • Zoned datetime: Temporal.ZonedDateTime, only if needed
  • No public Date values unless unavoidable for browser interop.
  • No dayjs or other date library.
  • Use Intl.DateTimeFormat for localized display.
  • Keep parsing and formatting in @energinet/watt/next/core/temporal.
  • Load Temporal polyfill only when globalThis.Temporal is missing.

13. Browser API Replacements

Use browser-native APIs instead of third-party runtime dependencies.

  • Angular Material button: native button.
  • Angular Material checkbox/radio: native input semantics plus token styling.
  • Angular Material dialog: native dialog.
  • Angular CDK overlay: Popover API and owned fallback.
  • Popper.js: Popover API and owned positioning fallback.
  • Maskito: native input handling and validation.
  • dayjs: Temporal and Intl.
  • Material table: semantic table.
  • Material tabs/stepper: native buttons, ARIA patterns, roving tabindex.
  • Toast library: signal service and ARIA live regions.
  • Drag/drop library: Drag and Drop API and File API.

14. Component Implementation Rules

Use modern Angular APIs

  • input()
  • output()
  • model()
  • computed()
  • signal()
  • ChangeDetectionStrategy.OnPush
  • Standalone components
  • Native control flow syntax

Avoid

  • @Input
  • @Output
  • @HostBinding
  • @HostListener
  • ngClass
  • ngStyle
  • ControlValueAccessor
  • Angular Material
  • Angular CDK
  • Use host metadata instead of @HostBinding and @HostListener.
  • Keep components small and focused.
  • Use tokens for all colors, spacing, shadows, radius, typography, and states.

20. Migration Strategy

  • Keep v1 stable while v2 is developed.
  • Migrate apps component by component.
  • Replace imports from @energinet/watt/button.
  • Use imports from @energinet/watt/next/button.
  • Do not replace @energinet/watt v1 entrypoints until v2 reaches parity.
  • Publish next as alpha/beta while Signal Forms remain experimental.

Promote v2 to the main package only after

  • Required components are complete.
  • Storybook parity is documented.
  • Accessibility checks pass.
  • Applications have migrated key flows.
  • Signal Forms API risk is accepted or stabilized.

21. Risks And Mitigations

  • Risk: Angular Signal Forms are experimental. Mitigation: ship next as alpha/beta and isolate form APIs.
  • Risk: removing CDK means owning accessibility behavior. Mitigation: add shared focus, keyboard, overlay, and scroll primitives.
  • Risk: Popover API behavior differs by browser. Mitigation: add browser tests and approved polyfill/fallback.
  • Risk: Temporal is not fully available in Safari. Mitigation: load Temporal polyfill only when missing.
  • Risk: Figma variable naming can drift. Mitigation: use token checks and PR automation.
  • Risk: token aliases and modes can be complex. Mitigation: normalize and snapshot generated token JSON.
  • Risk: one component per PR may be slow. Mitigation: prioritize primitives and high-use form controls first.

22. Definition Of Done For Watt Next Foundation

  • @energinet/watt/next/* entrypoints build.
  • Storybook shows Watt Next foundations.
  • Figma token sync works from Enterprise Variables API.
  • Generated token files are committed.
  • Token drift check runs in CI.
  • Temporal polyfill setup is available.
  • Shared primitives are available.
  • First simple component can be implemented without v1 runtime dependencies.

23. Definition Of Done For A Component

  • It uses an @energinet/watt/next/* entrypoint.
  • It does not import v1 Watt, Material, or CDK.
  • It uses generated design tokens only.
  • It matches documented Figma variants and states.
  • It matches required v1 Storybook features.
  • It supports Signal Forms if it is a form component.
  • It passes unit tests.
  • It passes Storybook interaction/a11y checks where relevant.
  • It is documented in Storybook.
  • It has keyboard and screen reader behavior documented.
  • It passes package and Storybook build.
